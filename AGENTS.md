# 项目智能体指南（AGENTS.md）

本项目采用 **AgentOS** 结构。**真理源（canonical truth）只有一份**，放在 `.agent/`。
各 IDE 的目录（`.kiro/`、`.cursor/`、`.claude/`、`.antigravity/`…）只是**薄壳（shim）**，通过引用拉同一份。

> 如果你在切换 IDE，**不需要迁移知识**——只需要确认对应 IDE 的 shim 目录是否存在。

## 语言约定（重要）

人机共读的内容默认使用**中文**。具体分工：

- **中文为主**：`README.md`、`AGENTS.md`、`.agent/core/`、`.agent/domain/`、`.agent/adr/`、
  `.agent/flows/`、`.agent/sessions/`、各 `_index.md`。
- **英文为主**：`.agent/sub-agents/*`（agent prompt + I/O schema）、`.agent/skills/*`
  （技术 reference）、`.agent/map/*`（自动生成）。
- **永远英文**：代码、文件名、字段名、commit message 的类型前缀（`feat:`、`fix:`、`chore:`）。

## 会话开始时的必读清单

按顺序读完这几项，**然后说"准备就绪"再开始任何动作**：

1. `.agent/skills/agent-discipline.md`——**Always-on 行为约束**（基于 Karpathy
   pitfalls + 项目扩展），是后续所有行为的护栏
2. `.agent/core/philosophy.md`——产品哲学与可判断的设计原则（P1、P2…）
3. `.agent/core/conventions.md`——代码、命名、分层规则
4. `.agent/core/boundaries.md`——模块边界（什么能 import 什么）
5. `.agent/map/architecture.md`——自动生成的项目地图（**如果 source-hash 失配，先跑
   `.agent/workflows/regenerate-map.md`**）
6. `.agent/adr/_index.md`——架构决策清单
7. `.agent/flows/_index.md`——前端用户流程清单（前端任务必读）

## 当用户请求工作时

1. **首次接入** AgentOS 到任何 repo：跑 `.agent/workflows/wizard.md`——它会
   判断空 repo / 既有项目 / 学习项目并分流。
2. 判断这个工作落在哪一层（参考 `.agent/core/boundaries.md`）。
3. **新功能**：跑 `.agent/workflows/start-feature.md`。
4. **架构调整**：跑 `.agent/workflows/architecture-change.md`。
5. **前端流程改动**（页面、组件、交互、跳转）：跑 `start-feature` 并产出/
   更新对应 `.agent/flows/{NNN}-*.md`。
6. **走查反馈 / bug / 报错 / 问题清单**：跑 `.agent/workflows/fix-review-feedback.md`。
   识别信号：用户提到"bug"、"报错"、"不对"、"修"、"问题"、粘贴错误信息、
   给出编号清单、截图标注问题。
7. **继续之前的任务**：跑 `.agent/workflows/resume-session.md`。
8. **接近上下文上限**：在 70% 时主动跑 `.agent/workflows/write-handoff.md`。
9. **空 repo 直接 0→1**（已知不需 wizard）：跑 `.agent/workflows/bootstrap-project.md`。
10. **既有项目嵌入**（已知不需 wizard）：跑 `.agent/workflows/retrofit-project.md`。
11. **`.agent/` 自身在膨胀**：跑 `.agent/workflows/prune.md`（epoch 末尾或触
    达阈值时）。

## 硬性规则（不得违反）

- **决策落到 ADR 或 flow，不留在对话里**。
  - 架构决策 → 新建 `.agent/adr/{NNNN}-{slug}.md`，引用 `_template.md`。
  - 前端流程 / 交互决策 → 新建或更新 `.agent/flows/{NNN}-{slug}.md`。
  - 在代码注释、PR 描述里引用 ID（如 `ADR-0007`、`FLOW-003`）。
- **架构调整必须先做影响分析**，调用 `impact-analyzer` sub-agent。
- **长任务必须在 70% 上下文时产出 `handoff.md`**，不要等到溢出。
- **自动生成的文件人类只读**。`.agent/map/` 下文件带 `Generated by:` 头部，**不要手编辑**。
- **不允许跨层 import**。下层（tokens、primitives）不得引用上层（features、pages）。
  由 `boundaries.md` 强制。
- **使用逻辑时间，不用 wall-clock**。项目内部引用用 ADR 编号、ULID 会话 ID、epoch
  tag、commit hash。Wall-clock 仅用于审计。
- **在 Antigravity 下运行时必须对齐 Native Planning 模式**：
  - Spec 生命周期映射：`implementation_plan.md`（分析设计）、`task.md`（开发 Checklist）和 `walkthrough.md`（Post-spec 走查总结）。
  - 在 Handoff / Resume 时，执行 `bash scripts/antigravity-sync.sh <backup|restore>`（全自动寻址，无需参数）。
- **Antigravity 软 Hook 协议**（无原生 Hook 时的替代）：
  - **任务开始前**：主动读 `philosophy.md` + `boundaries.md`（替代 pre-task-context-check）。
  - **ADR 创建后**：主动调 `impact-analyzer`（替代 post-adr-impact hook）。
  - **任务完成后**：主动调 `executor` 跑验证 + 检查 map 是否需更新（替代 post-task-verify）。
  - **Commit 前**：由 `lefthook` 自动执行 shim 同步检查 + 备份。

## Sub-agent 触发规则（硬性，不可跳过）

> **核心原则**：任何预期输出 > 200 行、或需读 > 10 个文件、或需外部资料的任务，
> **必须**调 sub-agent。主 agent 不得在主 context 里默默干这些事。

### 触发决策表（if → then）

| 你即将做的事　　　　　　　　　　　　　　　　| 调谁　　　　　　　　| 为什么　　　　　　　　　　　　　　　　　　 |
| ---------------------------------------------| ---------------------| --------------------------------------------|
| 搜索 / 理解 > 5 个文件的代码　　　　　　　　| **explorer**　　　　| 避免 grep 结果污染主 context　　　　　　　 |
| 读外部 docs / API / 库源码 / RFC　　　　　　| **researcher**　　　| 外部内容 token 巨大，只要结论　　　　　　　|
| 评估一个改动的影响范围　　　　　　　　　　　| **impact-analyzer** | 扫 dep graph 产出大量中间数据　　　　　　　|
| 跑 `npm test` / `cargo test` / 任何测试套件 | **executor**　　　　| 测试日志 3000+ 行，只要 pass/fail　　　　　|
| 跑 `npm run build` / `pnpm build` / 编译　　| **executor**　　　　| build 日志同理　　　　　　　　　　　　　　 |
| 跑 migration / 重 IO 脚本　　　　　　　　　 | **executor**　　　　| 同理　　　　　　　　　　　　　　　　　　　 |
| 批量 rename / codemod / lint-fix > 5 文件　 | **fixer**　　　　　 | 80 个文件的 diff 不该进主 context　　　　　|
| 验收一个 spec（unit + e2e + PBT 全套）　　　| **verifier**　　　　| 多 suite 结果需要结构化解读　　　　　　　　|
| UI 改动后看视觉效果　　　　　　　　　　　　 | **visual-reviewer** | 需要截图 + 多模态 + 状态矩阵　　　　　　　 |
| merge 后刷新 map / 索引　　　　　　　　　　 | **doc-syncer**　　　| 机械性更新，不需要主 agent 判断　　　　　　|
| 上下文 ≥ 70% / 阶段结束 / 用户暂停　　　　　| **scribe**　　　　　| 写 handoff 是 scribe 的活，不是主 agent 的 |

### 调用格式（必须可见）

```
🔧 Calling sub-agent: {name}
📥 Input: {结构化输入，匹配该 sub-agent 的 input contract}

[执行]

📤 Sub-agent {name} output: {结构化输出}
```

**禁止**：默默在主 context 里做 sub-agent 该做的事（违反 `agent-discipline` 6.10）。

### 不需要调 sub-agent 的情况

- 读 1–3 个文件 → 自己读
- 改 1–5 个文件 → 自己改
- 输出 < 50 行的命令 → 自己跑
- 不涉及外部资料的简单问题 → 自己答

### 详细 I/O 契约

见 `.agent/sub-agents/_index.md` 和各 sub-agent 独立文件。

### Antigravity 原生子代理映射

在 Antigravity IDE 中，可优先使用原生子代理替代 role-play：

| AgentOS sub-agent | Antigravity 原生替代 | 策略 |
|---|---|---|
| explorer | `research` 原生子代理 | 代码导航优化，比 role-play 更高效 |
| researcher | `research` 原生子代理 | 外部文档研究 |
| visual-reviewer | `browser` + Chrome DevTools MCP | 真实浏览器截图和交互测试 |
| executor | 后台任务（`run_command` + `manage_task`） | 测试日志通过异步获取 |
| fixer / doc-syncer | `self` 原生子代理 | 批量机械性任务适合克隆体并行 |
| scribe | `self` 原生子代理 | handoff 写作可委托克隆体 |

原生子代理不可用时回退到 role-play（按 6.10 规则可见地输出）。

## 跨域契约

跨域共享的内容放在 `shared/`：

- `shared/tokens/`——设计 token（DTCG 格式）
- `shared/schemas/`——运行时校验 schema（zod / proto）
- `shared/events/`——事件契约
- `shared/api-contracts/`——API 表面（OpenAPI / tRPC）

**永远引用，永远不内联具体值**。

## 产物去向

| 你产出了... | 写到... |
|---|---|
| 架构决策 | `.agent/adr/{NNNN}-{slug}.md` |
| 前端流程 / 交互定义 | `.agent/flows/{NNN}-{slug}.md` |
| 会话末尾的 handoff | `.agent/sessions/{ulid}/handoff.md` |
| 重新生成的图或汇总 | `.agent/map/{name}.{ext}` |
| 代码 | `apps/`、`packages/` |
| 可复用契约 | `shared/{tokens|schemas|events|api-contracts}/` |
| 可复用的 agent skill | `.agent/skills/{topic}.md` |

## 功能"完成"的定义

- 代码已合并
- `shared/` 中相关契约已更新
- 决策落到 ADR 或 flow
- 单测、PBT、e2e 已通过
- UI 改动有视觉 review 记录在 spec 中
- `.agent/map/` 已重新生成
- spec 状态置为 `done`（或归档若已废弃）
