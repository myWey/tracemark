# AgentOS——这套结构是怎么工作的

> 看一次就够。大部分会话不需要再读这份。

## 设计目标

1. **IDE 无关的真理源**。知识住在这里，IDE 配置都是薄壳。
2. **洋葱式稳定性**。内层（philosophy、domain）少动，外层（spec、code）常动。
   缓存住里面、替换掉外面。
3. **机器可消费**。每个 agent 要读的文件都有结构：front matter、稳定 ID、引用而非
   拷贝。
4. **逻辑时间**。ADR 编号、ULID、epoch tag、commit hash。wall-clock 只用于审计与漂移。
5. **自维护**。hook 重新生成 map；sub-agent 更新派生文档。人维护内层，agent 维护外层。

## 五层 + 前端流程层

```
层               目录                   稳定度    维护者
--------------   --------------------   -------   ------------------------
哲学             core/                  极高      人主导，agent 协助
领域             domain/                高        人 + agent
架构决策         adr/                   中        agent 在 PR 中起草，人 review
前端流程         flows/                 中        人主导，agent 起草 + 维护
契约             ../shared/             中        agent 在 feature 中维护
地图（派生）     map/                   低（生成）agent 通过 hook
会话             sessions/              短期      agent（scribe sub-agent）
```

外加三个横切区：

- `skills/`——按需加载的知识，agent 看任务相关性才拉
- `workflows/`——主 agent 可执行的可复用 prompt
- `sub-agents/`——专门 agent 的定义

## 真理源对照表

| 问题 | 看哪里 |
|---|---|
| 这个产品在做什么？ | `core/philosophy.md` |
| 这里词怎么用？ | `core/glossary.md` |
| 代码怎么写？ | `core/conventions.md` |

| 有哪些核心实体？ | `domain/entities.md` |

| 项目当前长什么样？ | `map/*`（不新鲜就重新生成） |

| 跟 harness 前沿对齐情况？ | `_HARNESS.md` |

## 文件头部约定

人写的文件起头：

```markdown
---
id: {stable-id}        # 如 philosophy、glossary、adr-0007、flow-002
status: draft|active|superseded
audience: human|agent|human+agent     # 谁该读
purpose: {一句话}                      # 为什么存在
last-confirmed: {YYYY-MM-DD}           # 人类最后一次确认"这条还成立"的日期
supersedes: [adr-0003]                  # ADR / flow 用
---
```

自动生成的文件起头：

```markdown
<!-- Generated by: .agent/workflows/regenerate-map.md
     At: {ISO 时间戳，仅供参考}
     Source-hash: {sha256}      ← 这才是判断是否过期的依据，不看时间
     Sources: [{globs}] at commit {hash} -->
```

## 增/改文件的约定

- 新 ADR → 复制 `adr/_template.md`，从 `adr/_index.md` 取下一个编号，更新索引。
- 新 flow → 复制 `flows/_template.md`，从 `flows/_index.md` 取编号，更新索引。
- 新 skill → 在 `skills/_index.md` 注册；agent 只加载已注册的 skill。
- 新 workflow → 放在 `workflows/`，名字用动词短语。
- 新 sub-agent → 在 `sub-agents/_index.md` 注册，独立文件里写 I/O 契约。
- 新顶层文档 → **必须更新 `_INDEX.md`**，否则就是孤儿。
- 地图文件 → 不要手编辑。跑生成器 workflow。

## 这套不是

- 不是项目管理系统。**不要在这里追 sprint / 日期**。用 git tag + epoch。
- 不是个人草稿本。临时笔记放 `sessions/{ulid}/notes.md`，不要塞进 `core/`。
- 不是无限的。文件 > 300 行就拆。agent 读得慢，人也跳。

## 语言约定

见 `AGENTS.md` 的"语言约定"小节。**默认中文，agent 内部接口英文，代码英文**。

## 文档体系自身的健康度

定期跑 `_AUDIT.md` 检查文档体系本身是否还遵守 12 条公认原则。这是范式的
"自体免疫"——文档不审计，几个月后会和代码一起熵增。
