# Spec-Driven Development

## When to Invoke

- A new feature spans > 2 files or > 50 lines.
- Introducing a new system or module.
- Requirements are ambiguous, or admit multiple reasonable interpretations.
- When you are about to ask "what do you actually want?".

## When NOT to Invoke

- Bug fixes under 50 lines (use the debugger subagent instead).
- Trivial config or style changes.
- Identical tasks already performed repeatedly in the current codebase.

## Core Principle

A spec is a 1-2 page structured document. It is not a novel, and it is not a user story.
It is a contract between human and agent.

**This skill does not generate spec files itself.** Instead, it guides the user to
invoke the Trae native `/spec` command. The native `/spec` command automatically
generates `spec.md` + `tasks.md` + `checklist.md` under the `.trae/specs/<change-id>/`
directory.

## File Structure Generated by the Native `/spec` Command

After running `/spec`, Trae generates the following under `.trae/specs/<change-id>/`:

```
.trae/specs/<change-id>/
├── spec.md        # Requirements spec (What / Why / Scope / Success Criteria)
├── tasks.md       # Decomposed task list
└── checklist.md   # Acceptance checklist (for code-review to verify against)
```

### `spec.md` Target Structure (6 Sections)

After the native `/spec` generates initial files, restructure `spec.md` to
match this framework. This integrates PRD-level problem definition with
technical implementation spec into a single document.

```markdown
# Spec: [Feature Name]

## Problem
- **Why It Matters**: [Business value in 1-2 sentences. Who hurts without this?]
- **Current State**: [What exists today. What's broken or missing.]

## Solution
- **Design Decisions**: [Key design choices and their rationale.]
- **Trade-offs**: [What alternatives were considered and why rejected.
  At least 2 alternatives with one-sentence rejection reason each.]

## Success Criteria
- [ ] [Verifiable criterion — each must map to an executable check]
- [ ] [e.g., "GET /users returns 200 and a valid user list"]
- [ ] [e.g., "rate limit enforced at 100 req/min"]

## Out of Scope
- [Explicitly excluded — prevents scope creep]
- [What this spec does NOT solve]

## Implementation Phases
- Phase 1: [Foundation] → verify: [check]
- Phase 2: [Core logic] → verify: [check]
- Phase 3: [Interface/Polish] → verify: [check]

## Constraints
- **Always**: [Must follow — e.g., "All API responses use {code, data, message}"]
- **Ask**: [Needs confirmation — e.g., "Changing database schema"]
- **Never**: [Forbidden — e.g., "No new dependencies without approval"]
```

### Writing Principles

- **Problem section is not optional.** This is what replaces a separate PRD.
  If you can't state why it matters, the feature isn't ready to spec.
- **Trade-offs must include at least 2 alternatives.** "We chose X because Y"
  is not enough. "We considered A, B, and X; rejected A because..., rejected
  B because..., chose X because..." is the minimum.
- **Success Criteria must be verifiable.** "Works correctly" is not a criterion.
  "Returns HTTP 200 with valid JSON body" is. Each criterion should have a
  corresponding checklist.md item.
- **Out of Scope is as important as In Scope.** It stops the agent from
  "helpfully" adding things nobody asked for.
- **Constraints use three tiers**: Always (enforced), Ask (needs human
  confirmation), Never (forbidden). This gives the agent clear boundaries.
- **Keep it under 2 pages.** A spec longer than 2 pages is trying to be a
  plan. Split: spec owns "what/why", plan owns "how".

## Spec Update Strategy (Living Document)

Specs are **living documents**, not one-shot artifacts. Update them as
implementation reveals new information:

| Trigger | Action | File |
|---------|--------|------|
| Bug fix | Update the relevant requirement in spec.md, note the bug and fix date | spec.md |
| Acceptance walkthrough finds issue | Record in checklist.md, tick after fix | checklist.md |
| Requirement change | Add Delta record at bottom of spec.md (do NOT delete original) | spec.md |
| Architecture change | Create new ADR, update ARCHITECTURE.md, reference from spec.md | docs/adr/ |
| Scope expansion | Explicitly amend spec.md, get re-approval before continuing | spec.md |

### Delta Record Format

When requirements change after approval, append to spec.md:

```markdown
## Delta: [Date] — [Change Summary]
- **Original**: [what the spec said before]
- **Changed to**: [new requirement]
- **Reason**: [why the change was needed]
- **Approved by**: [who approved the change]
```

## Workflow

1. **Clarify**: If the task is ambiguous, use `AskUserQuestion` to ask 1-3
   targeted questions. Do not guess. Do not assume.

2. **Research**: Delegate to an `explorer` subagent to learn:
   - Existing patterns in the codebase
   - Files that will be affected
   - Conventions to follow
   - Potential conflicts

3. **Trigger the native `/spec` command**: Guide the user to run `/spec`. The
   Trae native command automatically generates `spec.md` + `tasks.md` +
   `checklist.md` under `.trae/specs/<change-id>/`.
   - This skill does not directly Write spec files.
   - If supplementary information is needed, provide material in the
     conversation and let the native command integrate it.

4. **Confirm**: Show the generated spec content and request approval. If in
   plan mode, you may use `NotifyUser`. Do not enter the planning phase
   before approval is granted.

5. **Amend**: If the user requests changes, re-trigger `/spec` or describe
   the modifications in the conversation so the native command updates the
   files under `.trae/specs/<change-id>/`.

## Principles

- **The spec is a contract.** Once approved, implementation must stay in
  scope. Scope changes require explicit amendment.
- **Success criteria must be verifiable.** "Works correctly" is not a
  criterion. "Returns HTTP 200 with a valid JSON body" is.
- **Out of Scope is as important as In Scope.** It stops the agent from
  "helpfully" adding things nobody asked for.
- **Open Questions are not failures.** They surface unknowns early, when
  they are cheapest to resolve.
- **Keep it short.** A spec longer than 2 pages is trying to be a plan.
  Split them: spec owns "what/why", plan owns "how".

## Anti-Patterns

- Describing implementation steps in the spec (that is the plan's job).
- Vague success criteria ("should work well").
- No Out of Scope section (an open invitation to scope creep).
- Skipping research (leads to specs that ignore existing patterns).
- Entering implementation without explicit approval.
- This skill Writing spec files itself instead of guiding use of the
  native `/spec` command.
