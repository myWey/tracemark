# Code Review

## When to Invoke

- After implementation, before shipping.
- At every checkpoint (every 3-5 tasks).
- Before creating a PR.
- Anytime quality is uncertain.

## When NOT to Invoke

- Documentation-only changes (a lighter review suffices).
- Trivial config changes (visual inspection is enough).

## Core Principle

The reviewer must evaluate every item. "Looks good" is not a review.
The reviewer is adversarial: assume the code is wrong until proven correct.

**Key addition**: The review must verify against
`.trae/specs/<change-id>/checklist.md` item by item. This checklist is
auto-generated by the native `/spec` and `/plan` commands, and is the
contract bridge between spec/plan and code-review.

## Verify Against checklist.md

At the start of the review, first locate the checklist for the current
change:

```
.trae/specs/<change-id>/
└── checklist.md   # Acceptance checklist
```

### Verification Flow

1. **Identify the change-id**: Determine `<change-id>` from the git branch
   name, commit message, or the most recent `/spec` or `/plan` execution
   record.
2. **Read checklist.md**: Verify item by item whether the implementation
   satisfies each acceptance criterion.
3. **Mark each item**: For each checklist item, give PASS / FAIL / N/A,
   and record it in the review report.
4. **Unmet items are Critical Issues**: Any item in the checklist that
   does not pass is automatically classified as a Critical Issue in the
   review report, and must be fixed before ship.

### checklist.md Format Reference

```markdown
# Checklist: [Feature Name]

## Success Criteria (from spec.md)
- [ ] [verifiable criterion 1]
- [ ] [verifiable criterion 2]

## Implementation Steps (from tasks.md)
- [ ] [step 1 completed and verified]
- [ ] [step 2 completed and verified]

## Checkpoints
- [ ] Checkpoint 1: commit created, tests pass
- [ ] Checkpoint 2: commit created, tests pass
```

## The Review Checklist

The reviewer must evaluate every item. "Looks good" is not a review.

### Correctness
- [ ] Does the code do what the spec/plan says?
- [ ] Are edge cases handled? (empty input, null, overflow, concurrent access)
- [ ] Are error paths tested, not just the happy path?
- [ ] Are there off-by-one errors, race conditions, or resource leaks?

### Architecture
- [ ] Does it follow existing patterns in the codebase?
- [ ] Are module boundaries respected? (no cross-layer imports)
- [ ] Is the abstraction level appropriate? (not over-engineered, not too primitive)
- [ ] Does it introduce unnecessary coupling?

### Quality
- [ ] Is naming clear and consistent with conventions?
- [ ] Are functions < 40 lines with a single responsibility?
- [ ] No dead code, unused imports, or commented-out code blocks?
- [ ] No hardcoded secrets, no unnamed magic numbers?
- [ ] No leftover `console.log` / `print` / debug statements?

### Tests
- [ ] Do the tests test the right thing? (not just mock the wrong abstraction)
- [ ] If the implementation is wrong, would the test FAIL?
- [ ] Is coverage of new/changed code > 80%?
- [ ] Are both happy path and error path tested?
- [ ] Are test names descriptive? (`should reject expired tokens`)

### Safety
- [ ] No unguarded destructive operations?
- [ ] Input validation at system boundaries?
- [ ] No unjustified new dependencies?
- [ ] No committed secrets?
- [ ] No SQL injection, XSS, or path traversal vulnerabilities?

### Scope
- [ ] Only files listed in the plan were modified?
- [ ] No out-of-scope opportunistic refactors?
- [ ] No unrelated formatting changes?
- [ ] No new dependencies beyond what the plan specified?

### Checklist Cross-Check (added)
- [ ] Has `.trae/specs/<change-id>/checklist.md` been read?
- [ ] Has every checklist item been verified and marked PASS/FAIL/N/A?
- [ ] Are all Success Criteria satisfied?
- [ ] Have all Checkpoints' commit and test status been confirmed?

## Workflow

1. **Identify the change**: `git diff` (uncommitted) or
   `git diff <base>..HEAD`.

2. **Locate and read the checklist**: Find
   `.trae/specs/<change-id>/checklist.md` and verify item by item. If the
   checklist does not exist, note it in the report and return to require
   running `/spec` or `/plan` first.

3. **Delegate to a `reviewer` subagent** (read-only access):
   - The reviewer cannot fix issues — only report.
   - The reviewer is adversarial: assume the code is wrong until proven
     correct.

4. **Generate the review report**:

   ```markdown
   # Review Report: [Feature Name]

   ## Verdict: [PASS / NEEDS_CHANGES / FAIL]

   ## Checklist Cross-Check Results
   | Checklist Item | Status | Note |
   |----------------|--------|------|
   | [criterion 1] | PASS | — |
   | [criterion 2] | FAIL | [reason] |

   ## Critical Issues (must fix before ship)
   1. [file:line] [issue] → [suggested fix]

   ## Suggestions (should fix)
   1. [file:line] [issue] → [suggested fix]

   ## Nits (optional)
   1. [file:line] [issue]

   ## Passed Checks
   - [list of passed checklist items]
   ```

5. **Present the report** to the user.

6. **If NEEDS_CHANGES or FAIL**: Do not ship. Route fixes to an `executor`
   subagent, then re-review.

## The Critical Test Question

For every test, the reviewer must ask:

> **"If the implementation is wrong, what would make this test fail?"**

If the answer is "nothing" — the test is a false positive. It passes
regardless of correctness. This is the most common and most dangerous
failure mode in agent-generated tests.

Bad test example (false positive):
```javascript
// mocked the entire module, so the test always passes
jest.mock('./auth');
test('login works', () => {
  expect(auth.login()).toBe(true); // always true — tests nothing
});
```

Good test example:
```javascript
test('login returns token for valid credentials', async () => {
  const result = await login('user@example.com', 'correctpass');
  expect(result.success).toBe(true);
  expect(result.token).toMatch(/^eyJ/); // verify JWT format
});
```

## Principles

- **Adversarial by design.** The reviewer's job is to find problems, not
  to endorse. A review that finds nothing is suspicious, not successful.
- **Read-only.** The reviewer reports; the executor fixes. Separation of
  duties prevents self-justification.
- **Evaluate every item.** Skipping a checklist item because it "looks
  fine" is how bugs ship.
- **The verdict must be honest.** PASS means every item was checked.
  NEEDS_CHANGES means issues must be fixed. FAIL means there is a
  fundamental problem.
- **checklist.md is the contract.** It is generated by `/spec` and
  `/plan`, and is the single source of truth for acceptance. Any unmet
  checklist item blocks ship.

## Anti-Patterns

- Saying "looks good" without checking every item.
- The reviewer also fixing issues (no separation of duties).
- Tests mock the system under test (false positive).
- Skipping scope checks (lets opportunistic refactors ship).
- Reviewing only the happy path (bugs hide in error paths).
- Ignoring `.trae/specs/<change-id>/checklist.md` and judging acceptance
  by feel.
- Marking PASS when checklist items have not passed.
