# Plan Mode

## When to Invoke

- After a spec is approved (medium to large tasks).
- Even for small tasks, when multiple files are touched.
- Before any task where the architectural approach is critical.
- When the user says "let's plan first" or "how should we approach this?".

## When NOT to Invoke

- Single-file, under 50 lines, with an obvious approach.
- Tasks already performed identically in the current codebase.
- When the user explicitly says "just do it".

## Core Principle

A plan is a structured approach written before any code. It lists the files
to change, implementation steps, checkpoints, and risks. A wrong approach
executed well still produces the wrong thing.

**This skill does not generate plan files itself.** Instead, it guides the
user to invoke the Trae native `/plan` command. The native `/plan` command
automatically generates `spec.md` + `tasks.md` + `checklist.md` under
`.trae/specs/<change-id>/` (sharing the same change-id directory as `/spec`,
keeping spec and plan linked).

## File Structure Generated by the Native `/plan` Command

After running `/plan`, Trae generates/updates the following under
`.trae/specs/<change-id>/`:

```
.trae/specs/<change-id>/
├── spec.md        # Requirements spec (from /spec, serves as plan input)
├── tasks.md       # Decomposed task list (with implementation steps and checkpoints)
└── checklist.md   # Acceptance checklist (for code-review to verify against)
```

### `tasks.md` Format Reference

The native plan generates the task list with the following structure, as a
reference for "what the native plan produces":

```markdown
# Plan: [Feature Name]

## Approach
[2-3 sentences describing the architectural approach, and why it was chosen over alternatives.]

## Files to Modify

| File | Change | Risk |
|------|--------|------|
| `path/to/file.ts` | [what to change] | [Low/Medium/High — reason] |

## Files to Create

| File | Purpose |
|------|---------|
| `path/to/new.ts` | [why this file exists] |

## Implementation Steps

1. [Step — with verifiable expected output]
2. [Step — with verifiable expected output]
3. ...

## Checkpoints
- Checkpoint 1: after step [N] → commit, run tests
- Checkpoint 2: after step [M] → commit, run tests

## Risks & Mitigations
- **Risk**: [description] → **Mitigation**: [action]
- **Risk**: [description] → **Mitigation**: [action]

## Out of Scope
- [explicitly not doing X]
- [explicitly not doing Y]
```

## Workflow

1. **Read the spec** (if it exists) at `.trae/specs/<change-id>/spec.md`.

2. **Research the codebase**: delegate to a `planner` subagent to:
   - Find every file that will be touched (not just the obvious ones).
   - Identify existing patterns to follow.
   - Check for reusable existing utilities.
   - Identify risks, unknowns, and dependencies.

3. **Trigger the native `/plan` command**: Guide the user to run `/plan`.
   The Trae native command automatically generates `tasks.md` + `checklist.md`
   under `.trae/specs/<change-id>/`.
   - This skill does not directly Write plan files.
   - If supplementary information is needed, provide material in the
     conversation (e.g. file list, risk points) and let the native command
     integrate it.

4. **Show and confirm**: Present the generated plan and request approval.
   The human must check:
   - Does the approach fit the existing architecture?
   - Is the agent about to touch files that should not be modified?
   - Is it about to introduce an unwanted dependency?

5. **If the plan is wrong**: Correct it with specific clarifying information.
   Never let it "see what happens". A wrong plan executed well still produces
   the wrong thing.

6. **After approval**: Enter the execution phase. Follow the checkpoints in
   the plan.

## Principles

- **List every file.** A file change that surfaces during execution is a
  plan failure. If a new file needs to be changed during execution, stop
  and amend the plan.
- **Every step has verifiable output.** "Implement login" is not a step.
  "Create `login.ts` containing an `authenticate()` function returning
  `{success, token}`" is a step.
- **Checkpoints are mandatory.** They create rollback points. A problem
  found at checkpoint 2 of 5 is cheap; one found at the end is expensive.
- **Risks must be honest.** Do not hide risks to make the plan look
  cleaner. Surface them and let the human decide.
- **Out of Scope prevents scope creep.** The agent will "improve things
  along the way". The plan's Out of Scope section blocks this behavior.

## Anti-Patterns

- A plan that describes what to do but lists no files (too vague).
- A plan with no checkpoints (no rollback points).
- A plan that hides risks (leads to surprises during execution).
- Entering execution without explicit approval.
- Treating the plan as a suggestion rather than a contract.
- Scope changing during execution without amending the plan.
- This skill Writing plan files itself instead of guiding use of the
  native `/plan` command.
